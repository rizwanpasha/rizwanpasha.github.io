1. Creating a ( proxy ) Root Certificate Authority.

Root Certificate Authority or Root CA in short, is the authority who reserves the rights to provide SSL certificates for websites.
We can create a proxy CA to create self-signed SSL certificates for testing/personal use.
These certificates are based on x509 public-private key encryption standard.


IMPORTANT: Never user self-signed SSL certificates in the production environment.


Follow the below steps to generate Root CA.


i) First, let's generate a private key for our Root CA. Store this key in a secure location as this key will be used for generating further certificates and compromising the key can cause all of the certificates generated using this key to be invalidated or to exploit any security vulnerability.


openssl genrsa -des3 -out ROOT_CA.key 2048
genrsa : Uses RSA algorithm to create the private key.
-des3 : protects your private key password. use -nodes if you don't want to password protect the key, but it is highly recommended that you do.
-out : Name of the private key file.
2048 : Uses 2048 bit encoding.

Above command will prompt you to enter a password for the private key.


ii) Next, use the above generated private key to generate Root CA certificate.


openssl req -new -x509 -days 365 -key ROOT_CA.key -sha256 -out ROOT_CA.crt
-new : Generates new certificate.
-x509 : Defines standard for public key infrastructure
-days : Validity of certificate in days, here 365 days.
-key : Name of the private key generated in above step ie; ROOT_CA.key.
-sha256 : Hashing algorithm used in generating the certificate which is of 256 bits in length.
-out : Name of the generated certificate ie; ROOT_CA.crt.

Above command will prompt you to enter the password for the private key, followed by basic information which will be used to generate the certificate ie;


Country Name, State or Province Name,

Locality ( City name ),

Organization Name,

Organization Unit Name ( Can be department name ),

Common Name ( Commonly Name of the organization domain name, which will be used as an identifier ),

Email Address ( optional ).


This information can also be stored in a configuration ( .cnf ) file and used instead of manually entering them, Which will be demonstrated later.

2. Generating self-signed server certificate.

This method of generating certificate is applicable for most of the use-cases, where you have a domain name. Most common SSL certificates are server certificates, which is also called one-way SSL or single SSL.
This certificate is used by the browser to validate the server's identity and also to encrypt all of the communications between the server and the client devices.


Follow the below steps to generate Server certificate.


i) First, let's generate a private key for the server certificate.


openssl genrsa -des3 -out SERVER.key 2048
genrsa : Uses RSA algorithm to create the private key.
-des3 : Protects your private key password. use -nodes if you don't want to password protect the key, but it is highly recommended that you do.
-out : Name of the private key file.
2048 : Uses 2048 bit encoding.

Above command will prompt you to enter a password for the private key.


ii) Generate Certificate Signing Request ( CSR ) which will be used to generate self-signed server certificate using our proxy Root CA. Same CSR can also be used to get it signed by a valid CA.


openssl req -new -out SERVER.csr -key SERVER.key
-new : Generates new certificate.
-key : Name of the private key generated in above step ie; SERVER.key.
-out : Name of the generated CSR file ie: SERVER.csr.

Above command will prompt you to enter the password for the private key, followed by basic information which will be used to generate the certificate ie;


Country Name, State or Province Name,

Locality ( City name ),

Organization Name,

Organization Unit Name ( Can be department name ),

Common Name ( Commonly Name of the organization domain name, which will be used as an identifier ),

Email Address ( optional ).


iii) Generate a self-signed certificate for the server with the above generated CSR and Root CA.


openssl x509 -req -in SERVER.csr -CA ROOT_CA.crt -CAkey ROOT_CA.key -CAcreateserial -out SERVER.crt -days 365
-req : Requesting for a self-signed certificate.
-CA : Root CA's certificate file ie; ROOT_CA.crt.
-CAkey : Root CA's private key ie; ROOT_CA.key.
-CAcreateserial : Generates a unique id for every certificate created.
-out : Name of the generated certificate ie: SERVE.crt.
-days : Validity of certificate in days, here 365 days.

3. Generating self-signed server certificate for IP address and subdomains.

If you want to test sites running on IP addresses ( Usually during development stage or when the domain name is unavailable ) and registering subdomains this method is useful.

Follow the below steps to generate server-certificate for IP address.


i) First, let's generate a private key for the server certificate.


openssl genrsa -des3 -out SERVER.key 2048
genrsa : Uses RSA algorithm to create the private key.
-des3 : protects your private key password. use -nodes if you don't want to password protect the key, but it is highly recommended that you do.
-out : Name of the private key file.
2048 : Uses 2048 bit encoding.

Above command will prompt you to enter a password for the private key.


ii) Generate Certificate Signing Request ( CSR ) which will be used to generate self-signed server certificate using our proxy Root CA. Same CSR can also be used to get it signed by a valid CA.


openssl req -new -out SERVER.csr -key SERVER.key -config SERVER.csr.cnf
-new : Generates new certificate.
-key : Name of the private key generated in above step ie; SERVER.key.
-out : Name of the generated CSR file ie: SERVER.csr.
-config : Basic information used to generate CSR ( similar to the step in generating Root CA certificate ) can be also be saved in a file in the following format.

SERVER.csr.cnf file content, modiefy as per your need.

[ req ]

default_bits = 2048

prompt = no

default_md = sha256

distinguished_name = dn

req_extensions = req_ext

 

# change as per your need.

[ dn ]

C = IN

ST = Karnataka

L = Bangalore

O = THBS

OU = IT

# domain name.

CN = www.thbs.com

 

[ req_ext ]

basicConstraints = CA:FALSE

keyUsage = digitalSignature,keyEncipherment

extendedKeyUsage = serverAuth,clientAuth

subjectAltName = @alt_names

 

# useful for registering subdomains, and testing sites which are only running on IP address.

[ alt_names ]

# IMPORTANT: Certificates generated for IP address must also contain a named domain, So localhost will work fine.

DNS.1 = localhost

# Register multiple subdomains.

DNS.2 = subdomain_1.domain.com

DNS.3 = subdomain_2.domain.com

# Multiple IP addresses can also be registered.

IP.1 = 172.30.66.152

IP.2 = 127.0.0.1


Above command will prompt you to enter a password for the SERVER.key's private key.


iii) Generate a self-signed certificate for the server with the above generated CSR and Root CA.


openssl x509 -req -in SERVER.csr -CA ROOT_CA.crt -CAkey ROOT_CA.key -CAcreateserial -out SERVER.crt -days 365 -extfile SERVER_V3.ext
-req : Requesting for a self-signed certificate.
-CA : Root CA's certificate file ie; ROOT_CA.crt.
-CAkey : Root CA's private key ie; ROOT_CA.key.
-CAcreateserial : Generates a unique id for every certificate created.
-out : Name of the generated certificate ie: SERVER.crt.
-days : Validity of certificate in days, here 365 days.
-extfile : Contains information necessary to generate the certificate. The content of the SERVER_V3.ext file is as below.

SERVER_V3.ext file content, modiefy as per your need.

authorityKeyIdentifier = keyid,issuer

basicConstraints = CA:FALSE

keyUsage = digitalSignature, nonRepudiation, keyEncipherment, dataEncipherment

subjectAltName = @alt_names

 

# Must be same as SERVER.csr.cnf file

[ alt_names ]

DNS.1 = localhost

DNS.2 = subdomain_1.domain.com

DNS.3 = subdomain_2.domain.com

IP.1 = 172.30.66.152

IP.2 = 127.0.0.1


IMPORTANT: While registering multiple DNS and IP add them is serial order. If any of these options is not needed the don't add them.

Above command will prompt you to enter the password for the private key, followed by basic information which will be used to generate the certificate ie;

5. Creating new keystore from existing certificates.

Keystore is the easier way of managing and organizing all of the private keys and certificates. There are 2 formats PKCS12 ( Public Key Cryptography System ) and java keystore. PKCS12 is the new format provided by openssl, but java format is widely used.


i) Creating PKCS12 keystore.


openssl pkcs12 -export -clcerts -in CERTIFICATE.crt -inkey CERTIFICATE.key -out KEYSTORE.p12 -name certificate
pkcs12 : Format of the generated keystore.
-clcerts : Client certificates type.
-in : Input certificates to be added into keystore, ie; CERTIFICATE.crt.
-inkey : Private key file, ie; CERTIFICATE.key.
-out : Name for keystore to be generated, ie; KEYSTORE.p12.
-name : Unique alias of the certificate, Which is used to identify certificates in the keystore.

ii) Creating java keystore.


Java keystore is proprietary formats by Sunmicro systems, which is widely used. Currently there is no direct way of creating java keystore from existing certificates, but PKCS12 format keystore can be converted into Java format.


keytool -v -importkeystore -srckeystore CERTIFICATE.p12 -srcalias certificate -destkeystore CERTIFICATE.jks -deststoretype JKS
-v : Prints result in the readable format.
-importkeystore : Specifies that keystore format conversion operation to be performed.
-srckeystore : Name of the source keystore file.
-srcalias : Alias of the key to be imported into new keystore.
-destkeystore : Name of the destination keystore file.
-deststoretype : Type of the destination keystore format.

Above command can also be used to convert JKS format into PKCS12 format just by replacing -deststoretype into PKCS12


iii) PEM format for adding certificates into the browser.


Adding certificates to browsers need keystore to be in PEM format, ie; keystore with .pem extension. Which can be done with the following command.


openssl pkcs12 -in CERTIFICATE.p12 -out CERTIFICATE.pem -clcerts
-in : Input keystore file name.
-out : Output keystore file name.
-clcerts : Indicates certificate type.

6. Importing or add certificates into existing keystores.


keytool -importcert -file CLIENT.crt -alias client-certificate -keystore CLIENT_KEYSTORE.jks
-importcert : Indicates importing of certificates action.
-file : Name of the certificate file to be imported into keystore.
-alias : Alias given for the certificate.
-keystore : Name of the destination keystore.

7. Deleting the certificate from the keystore.


keytool -delete -alias client -keystore TRUSTSTORE.jks
-delete : Indicates the action to be performed, ie; deletion of the certificate from the keystore.
-alias : Alias of the certificate to be deleted from the keystore.
-keystore : Name of the target keystore file.

8. Renaming certificate alias.


keytool -changealias -keystore CLIENT_KEYSTORE.jks -alias client-certificate -destalias client-certificate-new
-changealias : Indicates the type of action to be performed, ie; renaming alias.
-keystore : Name of the target keystore file name.
-alias : Target alias which needs to be changed.
-destalias : New alias to be renamed with.

9. Changing keystore password.


keytool -storepasswd -keystore CLIENT_KEYSTORE.jks
-storepasswd : Indicates the type of action to be performed, ie; changing keystore password.
-keystore : Name of the target keystore whose password is to be changed.

10. List keystore content.


keytool -list -v -keystore KEYSTORE_NAME.jks
-list : Indicates the type of action to be performed, ie; listing the content of stores.
-v : Prints content on readable format for better understanding.
-keystore : Name of the target keystore whose content is to be listed.
